# 聚合AI工坊项目说明书

## 1. 项目概述

本项目是一个 AI 工具箱聚合网站，项目名为 `ai-toolbox-hub`，中文产品名为“聚合AI工坊”。

它的核心目标是：用户注册登录后，在网站中自行配置各大模型厂商的 API Key，然后通过统一的工具界面调用不同厂商、不同模型，完成写作、翻译、总结、代码审查、视觉提示词生成等 AI 工作流。

项目当前定位是 MVP 版本，重点不是做完整商业 SaaS，而是先把“用户自带 API Key + 多厂商模型统一调用 + 工具模板化 + 历史记录”这条主链路跑通。

已实现能力：

- 用户注册与登录
- PostgreSQL 数据库存储用户、密钥、历史记录
- 用户 API Key 加密保存，前端不接触明文
- 多模型厂商统一调用接口
- 工具模板驱动的工具广场
- 工具运行结果与历史记录保存
- 服务端 HTTP/HTTPS 代理支持
- 深色霓虹玻璃风格 UI
- Vitest 单元测试
- GitHub 仓库初始化与推送

## 4. 环境变量

本地 `.env` 不应该提交到 GitHub。项目已通过 `.gitignore` 排除 `.env`。

`.env.example` 提供示例：

```env
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/ai_toolbox_hub?schema=public"
AUTH_SECRET="replace-with-a-long-random-secret"
NEXTAUTH_URL="http://localhost:3000"
API_KEY_ENCRYPTION_SECRET="replace-with-32-byte-minimum-random-secret"
UPSTASH_REDIS_REST_URL=""
UPSTASH_REDIS_REST_TOKEN=""

# Optional server-side proxy:
# HTTPS_PROXY="http://127.0.0.1:7890"
# HTTP_PROXY="http://127.0.0.1:7890"
# ALL_PROXY="http://127.0.0.1:7890"
```

重要变量说明：

- `DATABASE_URL`：PostgreSQL 连接地址。
- `AUTH_SECRET`：NextAuth 使用的服务端密钥，生产环境必须使用强随机值。
- `NEXTAUTH_URL`：本地为 `http://localhost:3000`，部署后改为线上域名。
- `API_KEY_ENCRYPTION_SECRET`：API Key 加密密钥，长度至少 32 个字符，生产环境不能泄露或随意更换。
- `HTTP_PROXY` / `HTTPS_PROXY` / `ALL_PROXY`：可选。中国大陆或本地代理环境下，服务端 Node 请求模型厂商接口时通常需要配置。

当前本地曾检测到可用代理端口：

```text
127.0.0.1:7890
```

因此本地 `.env` 中已加入：

```env
HTTP_PROXY="http://127.0.0.1:7890"
HTTPS_PROXY="http://127.0.0.1:7890"
ALL_PROXY="http://127.0.0.1:7890"
```

相关接口：


注册接口会：

1. 校验昵称、邮箱、密码。
2. 检查邮箱是否已注册。
3. 使用 bcrypt 哈希密码。
4. 创建用户。

登录接口由 NextAuth Credentials Provider 处理。

## 9. API Key 加密机制

加密文件：

```text
lib/crypto/api-key.ts
```

加密方式：

```text
AES-256-GCM
```

关键点：

- 使用 `API_KEY_ENCRYPTION_SECRET` 生成 256-bit 密钥。
- 每次加密生成随机 `iv`。
- 保存 `ciphertext`、`iv`、`authTag`。
- 前端只显示 `keyHint`，不会返回明文 Key。

密钥保存接口：

```text
app/api/api-keys/route.ts
```

密钥删除接口：

```text
app/api/api-keys/[provider]/route.ts
```

安全边界：

- 明文 API Key 只在服务端短暂解密，用于调用厂商接口。
- 前端永远不接收明文 API Key。
- 数据库只保存密文。
- `.env` 不提交。

## 13. 工具运行流程

用户点击“运行工具”后的流程：

1. 前端收集：
   - `toolId`
   - `provider`
   - `model`
   - `inputs`
   - `temperature`
   - `maxTokens`

2. 调用接口：

```text
POST /api/tools/run
```

3. 后端检查登录态。

4. 后端限流：

```text
checkRateLimit(`tool:${userId}`, 20, 60_000)
```

5. 后端校验参数。

6. 根据 `toolId` 找到工具模板。

7. 检查必填字段。

8. 根据用户和厂商查找加密 API Key。

9. 解密 API Key。

10. 渲染 Prompt。

11. 调用对应模型厂商 adapter。

12. 成功后写入 `ToolHistory`。

13. 返回：

```json
{
  "id": "...",
  "output": "...",
  "tokenUsage": {},
  "latencyMs": 1234
}
```

## 14. 服务端代理机制

代理支持文件：

```text
lib/network/proxy.ts
```

背景：

在本地开发过程中，曾遇到 `api.deepseek.com` 被解析到 `198.18.x.x` fake-ip 网段。浏览器能走本机代理，但 Node.js 服务端 `fetch` 默认不会自动走系统代理，导致模型调用报：

```text
无法连接到模型厂商接口，请检查网络、代理或厂商服务状态。
```

解决方式：

- 安装 `undici`
- 使用 `ProxyAgent`
- 在模型请求前调用 `configureServerProxy()`
- 从环境变量读取：
  - `HTTPS_PROXY`
  - `HTTP_PROXY`
  - `ALL_PROXY`
  - 小写版本也支持

本地测试确认：

- `127.0.0.1:7890` 可用
- 通过代理请求 DeepSeek 能收到厂商返回的 `401 invalid key`
- 说明网络链路打通

## 15. 错误处理经验

### 1. Unexpected end of JSON input

曾出现：

```text
Failed to execute 'json' on 'Response': Unexpected end of JSON input
```

原因：

前端 `fetchJson()` 直接调用 `response.json()`，服务端返回空响应或非 JSON 时，浏览器会抛底层 JSON 解析错误。

修复：

- 前端先读取 `response.text()`
- 再安全 `JSON.parse`
- 非 JSON 错误显示文本摘要
- 空响应显示友好提示

相关文件：

```text
components/workspace-client.tsx
```

### 2. 服务端连不到模型厂商

曾出现：

```text
无法连接到模型厂商接口，请检查网络、代理或厂商服务状态。
```

原因：

Node.js 服务端没有走本机代理。

修复：

```text
lib/network/proxy.ts
```

### 3. Insufficient Balance

曾出现：

```text
Provider request failed: 402 {"error":{"message":"Insufficient Balance"}}
```

含义：

模型厂商账号余额不足或额度用完。

这不是代码问题，用户需要：

- 到厂商控制台充值
- 检查套餐额度
- 更换有余额的 API Key
- 切换到其它有余额的厂商模型

后端已经识别：

- `402`
- `Insufficient Balance`
- `insufficient_quota`
- `余额不足`

并返回友好中文提示。

## 19. 部署建议

当前项目设计时默认可部署到 Vercel，但需要准备外部 PostgreSQL。

部署前要配置：

```env
DATABASE_URL="线上 PostgreSQL 地址"
AUTH_SECRET="强随机密钥"
NEXTAUTH_URL="线上域名"
API_KEY_ENCRYPTION_SECRET="强随机密钥，至少 32 字符"
```

如果线上环境需要访问部分模型厂商，并且目标地区网络受限，也要配置：

```env
HTTP_PROXY=""
HTTPS_PROXY=""
ALL_PROXY=""
```

Vercel 注意事项：

- Vercel 本身通常不能直接使用本地代理。
- 如果模型厂商在部署地区不可访问，需要使用可在线访问的代理服务、网关服务，或选择厂商可访问的部署区域。
- 数据库建议使用 Neon、Supabase、Railway、Render、Aiven 等托管 PostgreSQL。

### Netlify 部署说明

本项目也可以部署到 Netlify。项目是 Next.js App Router 应用，包含 API Routes、认证和 Prisma，因此不是纯静态站点。Netlify 会自动使用 Next.js Runtime 将 SSR 和 API Routes 转换为 Netlify Functions。

仓库根目录已添加：

```text
netlify.toml
```

配置内容：

```toml
[build]
  command = "npm run build"
  publish = ".next"

[build.environment]
  NODE_VERSION = "22"

[dev]
  command = "npm run dev"
  targetPort = 3000
  port = 8888
  framework = "#auto"
```

Netlify 后台需要配置以下环境变量：

```env
DATABASE_URL="线上 PostgreSQL 连接串"
AUTH_SECRET="强随机密钥"
NEXTAUTH_URL="https://你的-netlify-域名.netlify.app"
API_KEY_ENCRYPTION_SECRET="至少 32 字符的强随机密钥"
```

如果部署后的 Netlify Functions 需要通过代理访问模型厂商，还需要配置：

```env
HTTP_PROXY=""
HTTPS_PROXY=""
ALL_PROXY=""
```

注意：Netlify 上不能使用本机 `127.0.0.1:7890` 这种代理地址，因为那只存在于本地电脑。线上代理必须是 Netlify Function 能访问到的公网代理或企业网关。

Netlify 部署步骤：

1. 登录 Netlify。
2. 选择 `Add new site`。
3. 选择 `Import an existing project`。
4. 授权并选择 GitHub 仓库：

```text
kenenlin/AI-aggregated-websites
```

5. Build command 填：

```text
npm run build
```

6. Publish directory 填：

```text
.next
```

7. 在 `Site configuration -> Environment variables` 中添加上面的环境变量。
8. 准备一个线上 PostgreSQL 数据库。
9. 第一次部署前或部署后，确保线上数据库已经建表。

当前项目还没有 Prisma migrations 目录。MVP 阶段可以临时用：

```bash
npx prisma db push
```

把 schema 推到线上数据库。执行前需要把本地 `DATABASE_URL` 临时设置为线上数据库地址。

更正式的生产做法是后续补 Prisma migration：

```bash
npx prisma migrate dev --name init
```

然后部署时使用：

```bash
npx prisma migrate deploy
```

不建议在 Netlify 每次 build 中自动执行 `prisma db push`，因为它会直接改线上数据库结构，不利于审计和回滚。

## 21. 已知限制

- 当前限流是内存限流，服务重启会清空，生产环境建议替换为 Upstash Redis 或 Vercel KV。
- 当前工具执行默认非流式返回。
- 当前图片理解工具只是生成多模态提示词，尚未实现图片上传和真实视觉模型调用。
- 当前没有完整 E2E 测试。
- 当前没有管理员后台。
- 当前没有平台统一计费系统。
- 当前服务端代理依赖环境变量，不同部署环境需要单独配置。

## 22. 维护者接手 checklist

接手项目时建议按顺序检查：

1. 确认 `.env` 是否存在。
2. 确认 PostgreSQL 是否启动。
3. 执行 `npx prisma db push`。
4. 执行 `npm run dev`。
5. 注册用户。
6. 添加厂商 API Key。
7. 确认本地代理是否需要配置。
8. 运行一个文本工具。
9. 若报余额不足，去厂商控制台充值或换 Key。
10. 提交前运行：

```bash
npm run lint
npm run typecheck
npm test
npm run build
```

## 23. 项目心智模型

这个项目可以理解成四层：

第一层是 UI 工作台：

```text
用户选择工具、厂商、模型、参数和输入内容
```

第二层是工具模板：

```text
把用户输入变成 Prompt
```

第三层是统一模型适配：

```text
把统一请求转换成各厂商 API 请求
```

第四层是安全与持久化：

```text
用户登录、API Key 加密、历史记录保存
```

核心价值在于：新增工具时优先改 `features/tools/templates.ts`，新增厂商时优先改 `features/providers/`，不要把工具逻辑和厂商逻辑硬塞进页面组件。
